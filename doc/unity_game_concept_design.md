# Unity 재개발용 게임 콘셉트 기획서

Date: 2026-05-24
Status: Unity engine development baseline.

## 문서 목적

이 문서는 `Conan: Labyrinths of the Serpent`를 Unity 엔진에서 실제 3D 1인칭
던전 크롤러로 개발하기 위한 기준이다. 과거 웹게임의 화면 구조, DOM,
LocalStorage, Three.js 구현, 인게임 에디터 패널을 그대로 옮기는 문서가 아니다.

목표는 웹 프로토타입에서 검증된 플레이 아이디어를 Unity의 런타임 구조와 Unity
Editor 기반 제작 파이프라인으로 재구성하는 것이다.

핵심 결정은 두 가지다.

- 게임은 플레이어용 독립 런타임 제품으로 만든다.
- 맵, 아이템, 몬스터, 스킬, 퀘스트, 이벤트 제작은 Unity Editor 도구에서 처리한다.

## 제품 분리 원칙

### Game Runtime

Game Runtime은 플레이어가 실행하는 완성 게임이다. 타이틀, 새 게임, 이어하기,
마을, 던전, 전투, 인벤토리, 퀘스트, 저장/불러오기만 포함한다.

Game Runtime은 에디터 UI, 제작 패널, 디버그용 생성기, 직접 데이터 편집 기능을
포함하지 않는다. 런타임은 빌드된 콘텐츠 번들과 검증된 정의 데이터만 읽는다.

### Unity Editor Tool

Editor Tool은 Unity Editor 안에서 실행되는 개발자/디자이너용 제작 도구다.
초기 형태는 EditorWindow, Custom Inspector, ScriptableObject, Scene View 도구로
구성한다.

Editor Tool은 다음 데이터를 만든다.

- 던전 층, 셀, 문, 계단, 배치, 조명, 재질
- 마을 허브와 NPC 배치
- 몬스터, 아이템, 스킬, 상점, 전리품
- 퀘스트, 이벤트, 조건, 보상, 상태 변화
- 테스트 플레이용 임시 콘텐츠 번들
- 릴리즈 런타임용 검증 완료 콘텐츠 번들

과거 웹 버전처럼 게임 안에 에디터 탭을 넣지 않는다. 제작은 Unity 엔진 안에서
이루어지고, 플레이어 빌드는 제작 도구를 포함하지 않는다.

### 교환 데이터

Game Runtime과 Editor Tool은 화면 코드를 공유하지 않고 데이터 계약만 공유한다.

- 콘텐츠 정의: 클래스, 스킬, 아이템, 몬스터, NPC, 퀘스트, 이벤트, 상점, 전리품
- 월드 정의: 마을, 던전 층, 셀, 문, 배치, 조명, 재질, 전투 배경
- 빌드 산출물: compiled map, content bundle manifest, validation report
- 런타임 저장: 플레이어 진행, 인벤토리, 퀘스트 상태, 방문 정보, 필드/전투 상태

런타임 저장 파일에는 제작 에디터 상태를 넣지 않는다. 에디터 프로젝트에는
플레이어 진행 상태를 넣지 않는다.

## 게임 정체성

`Conan: Labyrinths of the Serpent`는 검과 주술, 폐허와 금화, 저주와 생존을
중심에 둔 3D FPS 시점의 던전 크롤러 RPG다. 플레이어는 위험한 국경 도시에서
원정을 준비하고, 고대 뱀 신의 사원과 연결된 지하 미궁으로 내려간다.

이 게임은 빠른 액션보다 선택, 준비, 위치, 자원, 위험 관리가 중요한 던전
크롤러다. FPS 시점의 몰입감과 한 칸씩 전진하는 던전 크롤러의 긴장감, 미지의
방을 여는 불안, 전투 전후의 손실 관리, 탐욕과 생존 사이의 선택을 핵심 감정으로
둔다.

## 핵심 플레이 경험

- 어두운 던전에서 방향, 문, 함정, 소리, 단서를 읽으며 이동한다.
- Unity 3D FPS 카메라로 공간을 직접 보고 조작한다.
- 판정은 셀, 방향, 배치 데이터에 기반한 던전 크롤러 규칙으로 처리한다.
- 마을에서 장비, 치료, 스킬, 퀘스트, 보급을 준비한다.
- 던전에서는 한정된 체력, 회복 수단, 식량, 횃불, 상태 이상을 관리한다.
- 전투는 우연성과 선택이 섞인 주사위/릴 기반 턴제로 처리한다.
- 퀘스트와 이벤트는 단순 보상보다 위험, 저주, 평판, NPC 관계 변화를 만든다.
- 던전 탐험 결과는 마을 서비스, NPC 반응, 다음 원정 조건에 영향을 준다.

## 플레이어 판타지

플레이어는 거대한 군대의 지휘관이 아니라, 위험한 유적에 직접 들어가는 생존자다.
주인공은 강해질 수 있지만 언제나 상처, 저주, 자원 부족, 잘못된 선택의 대가에
노출된다.

동료는 대규모 파티 구성원이 아니라 특별한 관계와 리스크를 가진 제한된 협력자다.
동료는 전투력뿐 아니라 퀘스트, 언어 해독, 치료, 거래, 배신, 희생 같은 서사적
기능을 가진다.

## 기본 게임 루프

1. Title에서 New Game 또는 Continue를 선택한다.
2. New Game이면 캐릭터 생성 후 Town으로 진입한다.
3. Town에서 장비, 회복 수단, 스킬, 보급품을 준비한다.
4. Town의 임무 게시판 NPC에게 퀘스트를 1개 수주한다.
5. Town의 문/게이트 NPC와 상호작용해 활성 퀘스트의 Dungeon으로 이동한다.
6. Dungeon에서 방, 복도, 문, 함정, NPC, 이벤트를 탐색한다.
7. 몬스터 조우가 발생하면 Combat 씬으로 전환한다.
8. Combat 결과를 반영하고 Dungeon으로 복귀한다.
9. 더 깊이 들어갈지, 귀환할지, 휴식할지 선택한다.
10. Town으로 돌아와 보상 정산, 치료, 장비 교체, 새 단서 확보를 진행한다.
11. 실패, 사망, 주요 조건 달성 시 Ending으로 이동한다.
12. 누적 선택과 월드 상태가 다음 원정과 결말을 바꾼다.

## 씬 구성

기본 씬 구성은 다음 흐름을 따른다.

```text
Title
  -> New Game / Continue
  -> Town
  -> Dungeon
  -> Combat
  -> Dungeon
  -> Town / Ending
```

각 씬의 책임은 다음과 같다.

- `Title`: 새 게임, 이어하기, 설정, 종료
- `New Game`: 주인공 생성, 직업/배경/시작 보급 선택
- `Town`: 원정 준비, NPC 상호작용, 거래, 회복, 퀘스트 수주, 던전 출발
- `Dungeon`: 3D FPS 던전 탐험, 이벤트, 함정, 조우, 전리품
- `Combat`: 몬스터 조우 시 진입하는 주사위/릴 기반 턴제 전투
- `Ending`: 사망, 캠페인 결말, 주요 퀘스트 성공/실패

`New Game`은 독립 씬으로 둘 수도 있고 `Title`의 하위 화면으로 둘 수도 있다.
개발 카운트에서는 Title/New Game을 하나의 시작 영역으로 묶되, UI와 상태 전환은
분리해서 설계한다.

## 마을-퀘스트-게이트 구조

게임 시작 방식은 몬스터헌터식 허브 구조를 따른다. 마을은 플레이어가 원정을
선택하고 준비하는 안전 구역이며, 던전은 수주한 퀘스트를 수행하는 위험 구역이다.

기본 규칙은 다음과 같다.

- 플레이어는 마을의 임무 게시판 NPC에게 퀘스트를 받는다.
- 활성 퀘스트는 한 번에 1개만 가질 수 있다.
- 마을의 문/게이트는 NPC 판정 상호작용 대상으로 취급한다.
- 게이트는 활성 퀘스트가 있을 때만 해당 퀘스트의 던전으로 이동시킨다.
- 퀘스트가 없으면 게이트는 출발을 막고 임무 게시판 이용을 안내한다.
- 던전 클리어, 퀘스트 완료, 포기, 사망은 마을/게시판/저장 상태에 반영된다.

임무 게시판의 퀘스트 목록은 매번 변한다. 리롤 조건은 두 가지다.

- 타이머 기반 리롤
- 던전 클리어 후 리롤

초기 구현에서는 타이머를 실제 시간 기준으로 둘지, 게임 내 원정 횟수 기준으로 둘지
정해야 한다. 저장/불러오기 안정성을 생각하면 첫 단계는 `원정 완료 횟수`와
`마을 복귀 이벤트` 중심으로 구현하고, 실제 시간 기반 리롤은 이후 확장하는 편이
단순하다.

## 마을 NPC 서비스

마을은 원정 허브다. 마을 NPC는 메뉴가 아니라 월드 안에 배치된 서비스 제공자다.
플레이어는 NPC를 바라보거나 접근한 뒤 상호작용해 해당 NPC의 서비스를 연다.

현재 기준 마을에는 다음 NPC 서비스가 필요하다.

| NPC | 역할 | 필요한 시스템 |
| --- | --- | --- |
| 여관 | 체력 회복, 상태 회복 | 비용 지불, 회복 처리, 상태 이상 제거 |
| 훈련소 | 성장과 전투 훈련 | 성장 비용, 능력치/숙련도 갱신 |
| 대장간 | 장비 구매/판매 | 장비 데이터, 상점, 장비 비교, 장착 |
| 약재상 | 회복 아이템 구매/판매 | 소비 아이템, 상점, 인벤토리 |
| 기술 상인 | 스킬 구매/판매 | 스킬 데이터, 스킬 보유 상태, 스킬덱 |
| 학자 | 정보, 감정, 단서 제공 | 감정, 힌트, 퀘스트/월드 지식 |
| 임무 게시판 | 퀘스트 표시와 수주 | 퀘스트 목록, 활성 퀘스트 제한, 리롤 |
| 문/게이트 | 퀘스트 던전 출발 | 씬 전환, 활성 퀘스트 검사, 던전 선택 |

마을 NPC는 모두 같은 상호작용 인터페이스를 공유해야 한다. NPC별 차이는
상호작용 결과로 열리는 서비스 화면과 실행 가능한 액션으로 분리한다.

마을 구현 참고 파일은 [`webgame_ref/src/townRuntime.js`](webgame_ref/src/townRuntime.js)와
[`webgame_ref/src/data/npcs.json`](webgame_ref/src/data/npcs.json)이다.

## Unity 프로젝트 구조

권장 폴더 구조는 다음과 같다.

```text
Assets/
  Conn/
    Core/
    Runtime/
    Rendering/
    UI/
    Content/
    Editor/
    Tests/
doc/
Packages/
ProjectSettings/
```

### Core

순수 데이터 계약과 규칙을 둔다. 가능하면 MonoBehaviour 의존성을 피한다.

- ID 타입과 registry
- 콘텐츠 DTO
- 공식 계산
- 조건/보상/상태 이상 규칙
- 검증 규칙

### Runtime

플레이 세션과 게임 상태를 담당한다.

- 저장/불러오기
- 캐릭터, 인벤토리, 장비, 스킬덱
- 퀘스트와 월드 상태
- 마을 NPC 상호작용과 서비스
- 상점 거래와 재고
- 던전 탐험 상태
- 전투 상태와 턴 진행

### Rendering

Unity 씬, 프리팹, 머티리얼, 조명, 카메라 표현을 담당한다.

- 3D FPS 플레이어 카메라와 입력 연결
- 격자 맵 빌더
- 셀/벽/문/계단 프리팹 배치
- 던전 조명과 분위기
- 마을 허브 배치
- 전투 배경 표현

### UI

플레이어가 보는 화면만 담당한다.

- 타이틀
- 캐릭터 생성
- HUD
- NPC 대화/상호작용
- 상점 거래
- 퀘스트 게시판
- 자동 지도
- 인벤토리
- 장비 비교
- 스킬덱
- 전투 UI
- 퀘스트와 로그

### Editor

Unity Editor 전용 제작 도구를 둔다. 이 코드는 플레이어 빌드에 포함하지 않는다.

- 콘텐츠 편집창
- 던전 층 편집 도구
- 이벤트 그래프 편집 도구
- 검증 대시보드
- 콘텐츠 번들 빌드 도구
- 테스트 플레이 실행 도구

## Assembly 기준

초기부터 assembly definition을 나누어 런타임과 에디터 경계를 명확히 한다.

```text
Conn.Core
Conn.Runtime
Conn.Rendering
Conn.UI
Conn.Editor
Conn.Tests
```

`Conn.Editor`는 Editor 전용 assembly로 설정하고 release player build에 포함하지
않는다. Runtime 계층은 Editor assembly를 참조하지 않는다.

## 콘텐츠 제작 원칙

콘텐츠는 런타임 코드 안에 흩어지지 않고 데이터로 작성한다. Unity Editor Tool은
이 데이터를 검증하고, Game Runtime은 검증된 결과만 읽는다.

콘텐츠 단위는 다음처럼 나눈다.

- Definition: 스킬, 아이템, 몬스터, NPC, 퀘스트, 이벤트 같은 재사용 정의
- World: 마을, 던전 층, 방, 셀, 문, 배치, 조명, 재질
- Encounter: 전투 배치, 적 그룹, 보상, 전투 배경, 난이도
- Rule: 공식, 조건, 상태 이상, 보상 테이블, 생성 규칙
- Build Manifest: 어떤 콘텐츠와 월드가 특정 게임 빌드에 포함되는지 설명하는 목록

제작자는 ScriptableObject, Scene View 도구, 커스텀 인스펙터를 통해 콘텐츠를
관리한다. 런타임은 Addressables, Resources, StreamingAssets, 또는 자체 번들
중 프로젝트 상황에 맞는 방식으로 검증된 산출물을 읽는다.

## 주요 시스템 개념

### 탐험

탐험은 Unity 3D FPS 카메라와 격자 기반 던전 크롤러 판정을 결합한다. 플레이어는
시야를 자유롭게 둘러볼 수 있지만, 이동, 충돌, 문, 함정, 이벤트, 전투 진입은 셀과
방향 기준으로 판정한다.

던전은 방과 복도, 잠긴 문, 비밀문, 계단, 안전지대, 위험지대, 이벤트 지점,
전투 지점으로 구성된다. 자동 지도는 플레이어가 실제로 확인한 정보를 중심으로
갱신한다.

### 마을

마을은 원정 사이의 전략 허브다. 여관, 치료, 훈련, 장비 거래, 스킬 구매, 감정,
퀘스트 수주, 정보 수집을 제공한다.

마을은 단순 메뉴가 아니라, 던전 결과가 되돌아오는 장소다. NPC는 화면 메뉴가
아니라 월드에 배치된 서비스 제공자이며, NPC 생존, 퀘스트 결과, 평판, 저주, 보물
반입 여부에 따라 서비스 조건과 대사가 달라질 수 있다.

현재 기준의 마을은 여관, 훈련소, 대장간, 약재상, 기술 상인, 학자, 임무 게시판,
문/게이트의 8개 상호작용 지점을 가진다. 모든 지점은 NPC처럼 감지되고, 플레이어가
바라보거나 접근한 상태에서 상호작용 키를 누르면 해당 서비스 UI를 연다.

### 전투

전투는 별도 씬/상태 머신에서 진행되는 턴제 전투다. 던전 안에서 직접 싸우는
방식이 아니라 `Dungeon -> Combat -> Dungeon` 흐름이다. Dungeon에서 몬스터와
조우하면 Combat으로 전환하고, 승패/보상/상태 변화를 처리한 뒤 Dungeon 세션으로
복귀한다.

핵심은 명령 선택만이 아니라, 주사위/릴 결과를 어떻게 조합해 이번 턴을 구성할지
결정하는 데 있다.

기본 로직은 [`레거시_전투시스템.md`](레거시_전투시스템.md)에 정리된 웹게임
전투를 참고한다. 구현은 JavaScript를 포팅하는 방식이 아니라, 같은 규칙을 Unity
C#의 전투 도메인 모델과 UI로 재작성한다.

핵심 참조 파일은 다음이다.

- 전투 세션, 턴, 승패, 보상: [`webgame_ref/src/combatRuntime.js`](webgame_ref/src/combatRuntime.js)
- 주사위/스킬 릴 규칙: [`webgame_ref/src/diceCombatRuntime.js`](webgame_ref/src/diceCombatRuntime.js)
- 스킬 데이터: [`webgame_ref/src/data/skills.json`](webgame_ref/src/data/skills.json)

전투에서 사용하는 스킬/주사위 loadout 구조는 [`레거시_스킬시스템.md`](레거시_스킬시스템.md)를
참고한다.

전투 시스템은 다음 목표를 가진다.

- 무작위 결과가 매 턴 다른 선택을 만든다.
- 스킬덱 구성은 장기 전략을 만든다.
- 쿨다운, 상태 이상, 아이템 사용은 단기 전술을 만든다.
- 적 전열/후열, 보스 패턴, 도주, 보상, 패배 상태를 처리한다.

### 캐릭터와 성장

주인공은 직업, 배경, 능력치, 장비, 스킬덱으로 정의된다. 성장은 단순 수치 상승만이
아니라, 더 위험한 선택지를 감당할 수 있는 능력 확장으로 설계한다.

동료는 최대 1명 기준으로 설계한다. 동료는 독립적인 상태, 성향, 관계, 퀘스트
연결을 가진다. 동료의 사망, 이탈, 배신, 구출은 월드 상태로 남는다.

### 스킬과 스킬덱

스킬 시스템은 캐릭터가 보유한 스킬을 주사위 면에 장착하는 구조다. 캐릭터는
기본적으로 5개의 6면 주사위를 가지며, 각 면은 눈 값과 스킬 ID를 가진다. 전투
턴에는 각 주사위에서 나온 roll을 STOP으로 확정하고, 제한 개수만큼 선택해 효과를
해석한다.

스킬 쿨다운은 스킬 ID 전체가 아니라 `dieId::faceIndex` 슬롯 단위로 적용한다. 이
규칙은 같은 스킬을 여러 면에 배치했을 때 각 면이 독립적인 전술 자원이 되게 만든다.

Unity 구현에서는 `SkillDefinition`, `SkillInventory`, `DiceLoadout`,
`DiceFaceDefinition`, `SkillFormulaEvaluator`, `SkillEffectResolver`를 Core/Runtime
계층에 두고, 스킬덱 장착 UI와 기술 상인 UI는 UI 계층에 둔다.

### 인벤토리와 장비

인벤토리는 탐험 압박을 만드는 시스템이다. 장비, 소비품, 전리품, 저주받은 물건,
감정되지 않은 유물, 퀘스트 물품을 다룬다.

장비는 전투 성능뿐 아니라 이동, 함정, 협상, 해독, 저항, 특정 이벤트 선택지에
영향을 줄 수 있다.

### 퀘스트와 이벤트

퀘스트는 게시판식 목표와 던전 내부 사건을 모두 포함한다. 이벤트는 선택지, 조건,
보상, 피해, 상태 변화, NPC 관계 변화를 만든다.

좋은 선택지는 항상 명백한 선악이 아니라, 보상과 손실을 동시에 제시해야 한다.
하이보리아풍 톤에 맞게 탐욕, 생존, 배신, 저주, 고대 신앙, 용병 계약을 주요
동기로 사용한다.

퀘스트 게시판은 여러 후보 퀘스트를 보여주지만, 플레이어가 들고 갈 수 있는 활성
퀘스트는 1개뿐이다. 퀘스트는 대상 던전, 목표 조건, 보상, 실패/포기 정책, 게시판
노출 조건을 가진다. 게시판 목록은 타이머 또는 던전 클리어 후 리롤된다.

## 저장과 테스트 플레이

런타임 저장은 플레이어 진행만 포함한다.

- 캐릭터와 동료 상태
- 인벤토리, 장비, 스킬덱
- 퀘스트 진행
- 방문한 셀과 자동 지도 정보
- 월드 상태와 NPC 관계
- 현재 원정 상태

Editor Tool의 테스트 플레이는 별도 임시 세션으로 실행한다. 테스트 플레이 저장은
릴리즈 저장 슬롯을 오염시키지 않는다.

## Git 저장소 기준

원격 저장소는 `https://github.com/mephiblin/conn.git`을 사용한다.

Unity 프로젝트 루트를 저장소 루트로 사용한다. Git에 포함할 기본 대상은 다음과
같다.

- `Assets/`
- `Packages/`
- `ProjectSettings/`
- `doc/`
- `.gitignore`
- `.gitattributes`

Git에서 제외할 대상은 다음과 같다.

- `Library/`
- `Temp/`
- `Logs/`
- `UserSettings/`
- IDE 임시 파일
- 로컬 빌드 산출물

Unity 설정은 다음 상태를 유지한다.

- Version Control: Visible Meta Files
- Asset Serialization: Force Text
- 대형 바이너리 에셋: `git-lfs` 설치 후 LFS 관리

## 첫 Vertical Slice

초기 목표는 전체 게임의 축소판을 Unity에서 끝까지 동작시키는 것이다.

1. 타이틀
2. 캐릭터 생성
3. 마을 허브
4. 임무 게시판에서 퀘스트 1개 수주
5. 게이트 NPC로 원정 시작
6. 던전 한 층 탐험
7. 고정 조우 전투
8. 보상 획득
9. 마을 귀환과 퀘스트 정산
10. 게시판 퀘스트 리롤
11. 저장/불러오기

이 범위가 닫힌 뒤 Editor Tool의 제작 편의성을 확장한다.
